# Feature Specification: PakEcon.ai Platform Enhancement — v2.0

**Feature Branch**: `002-platform-enhancement`
**Created**: 2026-03-17
**Status**: Draft — Approved for Implementation
**Input**: User direction to implement all research-identified enhancements including critical fixes, Tier 1/2/3 tools, agent improvements (Groq/LLaMA 3, Agent D Social Publisher), technical improvements (PWA, Analytics, Search, Newsletter), and AdSense-optimized content strategy.

---

## Overview

This specification covers the complete second phase of PakEcon.ai development. It builds on the deployed v1.0 foundation and transforms the platform from a "ready to deploy" state into a genuine daily-use destination with sufficient content velocity and tool depth to achieve Google AdSense approval and sustain organic traffic growth.

**Mission**: Every Pakistani — taxpayer, farmer, freelancer, homebuyer, or diaspora worker — should find PakEcon.ai indispensable for at least one daily or weekly financial task.

## User Scenarios & Testing

### User Story 1 — Publisher Agent Actually Publishes Content (Priority: P0 — BLOCKER)

The agent swarm runs every 6 hours and generates market insights, but these insights are never committed to the repository because the Publisher Agent (Agent C) only logs "Would create commit" instead of calling the GitHub API.

**Why this priority**: Without real Git commits, the site generates zero new content. All content automation is non-functional. This blocks AdSense approval (requires 30+ pages) and defeats the entire project's content strategy.

**Acceptance Scenarios**:

1. **Given** Agent C generates a new MDX insight file, **When** it calls the GitHub API, **Then** the file appears in the repository under `src/content/insights/YYYY-MM-DD-slug.mdx` within 60 seconds
2. **Given** the `HUMAN_REVIEW_MODE` environment variable is `true` (first 30 days), **When** Agent C publishes, **Then** it creates a Pull Request instead of a direct commit to `main`
3. **Given** the `HUMAN_REVIEW_MODE` environment variable is `false`, **When** Agent C publishes, **Then** it commits directly to `main` and triggers Cloudflare Pages auto-deploy
4. **Given** Agent C commits successfully, **When** the commit lands in GitHub, **Then** it calls the Google Indexing API to request indexing of the new page URL
5. **Given** the GitHub API call fails (token expired, rate limit), **When** Agent C encounters the error, **Then** it logs the error to D1 `agent_logs` table and retries once after 30 seconds before marking the workflow stage as `error`

### User Story 2 — Scrapers Fetch Real Data from Official Sources (Priority: P0 — BLOCKER)

All four scrapers (SBP, PBS, FBR, Commodities) currently use placeholder URLs or mock data. The D1 database is never populated with real economic data.

**Why this priority**: Without real data, exchange rates are fictional, commodity prices are fictional, and every insight generated by the Analyst Agent is based on invented numbers. The platform's entire value proposition collapses.

**Acceptance Scenarios**:

1. **Given** the Scraper Agent runs, **When** it calls SBP EasyData, **Then** PKR/USD, PKR/EUR, PKR/GBP, PKR/AED, PKR/SAR, and PKR/CNY rates are saved to D1 `exchange_rates` table with today's date
2. **Given** the Scraper Agent runs, **When** it calls PBS for SPI data, **Then** weekly sensitive price indicators for at least 8 food items are saved to D1 `commodity_prices` table
3. **Given** the Scraper Agent runs, **When** it calls OGRA or PSOPK for petrol prices, **Then** petrol, diesel, and kerosene prices (PKR/litre) are saved to D1 `commodity_prices` table
4. **Given** a data source is unreachable (timeout > 15 seconds), **When** the scraper encounters the error, **Then** it falls back to the most recent D1 record for that source, logs a warning, and continues the workflow
5. **Given** the scraper successfully fetches data, **When** it saves to D1, **Then** it uses `INSERT OR REPLACE` (UPSERT) to avoid duplicate records for the same date

### User Story 3 — Live Electricity Bill Calculator (Priority: P1)

A household or business owner in Pakistan wants to estimate their WAPDA/DISCO electricity bill based on their monthly units consumed. They should see their bill breakdown by slab, applicable taxes (GST, fuel cost adjustment, fixed charges), and total payable amount.

**Why this priority**: Electricity bill checking is one of the highest-frequency financial pain points for Pakistani consumers. Bills arrive monthly, slabs change with each budget, and users frequently dispute bills. This tool will drive the highest repeat visits of any tool on the platform.

**Acceptance Scenarios**:

1. **Given** a user enters 350 units consumed in Karachi (KESC/K-Electric), **When** they calculate, **Then** the system displays bill breakdown: energy charges by slab, fixed charges, GST (17%), fuel price adjustment, and total payable in PKR
2. **Given** a user selects their DISCO from a dropdown (LESCO, HESCO, MEPCO, PESCO, QESCO, IESCO, GEPCO, SEPCO, K-Electric), **When** they calculate, **Then** the correct tariff structure for that utility is applied
3. **Given** a user selects "Commercial" or "Industrial" consumer type, **When** they calculate, **Then** commercial/industrial tariff rates are applied instead of residential rates
4. **Given** a user views the result, **When** the bill is displayed, **Then** they see a breakdown table: Slab 1 units/rate/cost, Slab 2 units/rate/cost (etc.), Fixed charges, GST, Fuel adjustment, Total — matching NEPRA tariff structure
5. **Given** a user enters 0 units, **When** they calculate, **Then** the system still shows minimum bill (fixed charges only) as applicable per NEPRA rules

### User Story 4 — Zakat Calculator (Priority: P1)

A Muslim Pakistani wants to calculate their annual Zakat obligation. They should be able to enter their assets (cash, gold, silver, savings, business goods) and liabilities, and the system should calculate their Zakat liability based on the current Nisab threshold (derived from live gold/silver prices in D1).

**Why this priority**: Zakat is a religious obligation for approximately 95% of Pakistan's population. Ramadan creates a massive annual traffic spike. This tool will attract extremely loyal repeat users and high engagement time.

**Acceptance Scenarios**:

1. **Given** a user visits the Zakat Calculator, **When** the page loads, **Then** the current Nisab threshold is displayed in PKR, calculated from today's silver price (612.36 grams of silver) fetched from D1
2. **Given** a user enters cash savings of PKR 500,000, gold of 50 grams, and no liabilities, **When** they calculate, **Then** the system displays: total zakatable wealth, whether Nisab threshold is met, and Zakat due (2.5% of total if above Nisab)
3. **Given** a user has debts/liabilities exceeding their assets, **When** they calculate, **Then** the system clearly states: "Your liabilities exceed your assets — no Zakat is due" with an explanation
4. **Given** a user wants to use gold Nisab (87.48 grams gold equivalent), **When** they toggle between Silver Nisab and Gold Nisab options, **Then** the calculation updates accordingly using the current gold price from D1
5. **Given** the D1 price data is more than 24 hours old, **When** the user loads the page, **Then** the system shows the last known price with a timestamp: "Based on price as of [date]"

### User Story 5 — Loan / EMI Calculator (Priority: P1)

A Pakistani user planning to take a home loan, car loan, or personal loan wants to calculate their monthly installment (EMI), total interest paid, and total amount payable. They should be able to choose between conventional (interest-based) and Islamic (Murabaha/diminishing Musharakah) financing modes.

**Why this priority**: Loan calculators drive some of the highest-CPC ad traffic in Pakistan. Banks and NBFIs aggressively bid for these keywords. The dual conventional/Islamic mode is a unique differentiator — most calculators only offer one mode.

**Acceptance Scenarios**:

1. **Given** a user enters PKR 5,000,000 home loan, 20% down payment, 20-year term, 21% annual interest (conventional), **When** they calculate, **Then** the system displays monthly EMI, total interest, total payable, and an amortization schedule (first 12 months + summary)
2. **Given** a user switches to "Islamic Financing" mode, **When** they calculate with same inputs, **Then** the system calculates using **Murabaha (cost-plus)** model: `totalProfit = loanAmount × (annualProfitRate/100) × tenureYears`; `monthlyInstallment = (loanAmount + totalProfit) / tenureMonths` — a flat profit schedule (not compounding); the label "Interest" is replaced with "Profit" throughout the UI
3. **Given** a user views the amortization schedule, **When** they see the table, **Then** each month shows: month number, opening balance, EMI/installment, principal component, profit/interest component, closing balance
4. **Given** a user wants to prepay, **When** they enter a "Lump Sum Prepayment" amount and month, **Then** the calculator shows new EMI or reduced tenure options
5. **Given** a user changes any input, **When** they recalculate, **Then** the result updates within 200ms (pure client-side computation)

### User Story 6 — Live Currency Converter (Priority: P1)

A user wants to convert between PKR and any major currency at the current interbank rate (sourced from SBP data in D1). They should also see a 30-day trend chart showing PKR movement against that currency.

**Why this priority**: Currency conversion is searched thousands of times daily by Pakistan's large remittance-receiving population and importers/exporters. This creates a daily-use hook that keeps users returning.

**Acceptance Scenarios**:

1. **Given** a user visits the Currency Converter, **When** the page loads, **Then** live exchange rates for USD, EUR, GBP, AED, SAR, CAD, AUD, CNY, and JPY are displayed (fetched from `/api/exchange-rates`)
2. **Given** a user enters PKR 50,000 and selects USD, **When** they convert, **Then** the result shows the equivalent in USD using the current SBP interbank rate, plus the date/time of the rate
3. **Given** a user selects a currency pair, **When** the conversion is displayed, **Then** a 30-day sparkline chart shows PKR trend against that currency using historical D1 data
4. **Given** a user clicks "Swap", **When** swap is applied, **Then** the from/to currencies are reversed and the calculation updates
5. **Given** a user views the open market rate vs interbank rate, **When** both are available in D1, **Then** both rates are displayed side-by-side with a note explaining the difference

### User Story 7 — Salary Slip Generator (Priority: P1)

A HR professional or employee wants to generate a professional-looking monthly salary slip (payslip) in PDF format. They should be able to enter basic salary, allowances (house rent, medical, conveyance, etc.), deductions (EOBI, SESSI, income tax advance, loan recovery), and generate a printable PDF — all client-side.

**Why this priority**: Pakistan's HR sector is heavily under-digitized. Free salary slip generators are extremely popular, especially in SMEs where HR software is too expensive. Generating a PDF in-browser (using existing pdf-lib.js dependency) adds zero server cost.

**Acceptance Scenarios**:

1. **Given** a user enters company name, employee name, designation, basic salary, and standard allowances, **When** they generate the slip, **Then** a professional PDF is downloaded with all earnings and deductions, gross salary, net salary, and a company/month header
2. **Given** a user enters income tax advance deduction, **When** the slip is generated, **Then** the tax amount auto-calculates from the Tax Calculator's FBR 2026 slabs (divided by 12 for monthly) and pre-fills the field; user can override
3. **Given** a user generates a slip, **When** the PDF downloads, **Then** no data has been uploaded to any server (verified by network tab showing zero POST requests)
4. **Given** a user wants to save their company template, **When** they click "Save Template", **Then** all fields are saved to `localStorage` and pre-filled on next visit
5. **Given** a user views the slip preview, **When** it renders, **Then** it is formatted for A4 printing with company logo placeholder, structured earnings/deductions table, and authorized signatory lines

### User Story 13 — Groq/LLaMA 3 in Analyst Agent (Priority: P1)

The Analyst Agent (Agent B) currently uses hardcoded string templates to generate "analysis." This produces repetitive, obviously templated content that will be detected as low-quality by Google's systems.

**Acceptance Scenarios**:

1. **Given** Agent B has scraped new data with delta > 1%, **When** it calls the Groq API with the M.Phil Economics analyst persona prompt, **Then** it receives a unique 300+ word economic analysis in professional tone
2. **Given** the Groq API is unavailable (rate limited or down), **When** Agent B encounters the error, **Then** it falls back to OpenAI GPT-4o-mini for that single request and logs: "GROQ_FALLBACK: used OpenAI for insight ID [x]"
3. **Given** Agent B receives the LLM response, **When** it validates the content, **Then** it checks: word count > 250, contains at least one numeric figure from the scraped data, and does not contain placeholder text like "[INSERT DATA]"
4. **Given** the generated content passes validation, **When** it is saved to D1 `market_insights`, **Then** the `generated_by` column stores `"groq:llama-3.3-70b-versatile"` or `"openai:gpt-4o-mini"` for audit trail
5. **Given** Agent B analyzes commodities data, **When** the insight is generated, **Then** the content references 2026 Pakistan-specific context: Digital Nation Act, IMF program status, CPEC developments — these are included in the system prompt

### User Story 14 — Agent D: Social Publisher (Priority: P1)

After Agent C publishes content to GitHub, a new Agent D automatically posts a summary to the site's Twitter/X account and Telegram channel — extending reach without manual effort.

**Acceptance Scenarios**:

1. **Given** Agent C successfully commits a new insight to GitHub, **When** Agent D triggers, **Then** it posts to Twitter/X within 5 minutes: title + 3 key numbers from the insight + site URL (under 280 characters)
2. **Given** Agent D posts to Telegram, **When** the message is sent, **Then** the Telegram message includes: full insight title, all key summary bullets, source citations, and a "Read more" link — formatted with Markdown
3. **Given** the Twitter API returns rate limit error (429), **When** Agent D encounters it, **Then** it queues the tweet in KV with a `retry_at` timestamp (1 hour later) and does not fail the workflow
4. **Given** `TWITTER_ENABLED=false` environment variable is set, **When** Agent D runs, **Then** it skips Twitter posting and logs "Twitter disabled — skipping" but still posts to Telegram
5. **Given** Agent D completes, **When** the workflow finishes, **Then** the D1 `market_insights` table updates the insight row with `twitter_posted=true`, `telegram_posted=true`, and respective post IDs

Every Monday at 08:00 PKT, the agent swarm generates a "Weekly Economic Digest" post summarizing all tracked indicators from the previous 7 days — even if no single indicator crossed the 1% delta threshold.

**Acceptance Scenarios**:

1. **Given** Monday 08:00 PKT cron triggers, **When** the Digest Agent runs, **Then** it queries D1 for all exchange_rates, commodity_prices, and market_insights from the past 7 days

3. **Given** the digest is published, **When** the MDX file is committed, **Then** the URL is `/insights/weekly-digest-YYYY-MM-DD` and the Schema.org type is `NewsArticle` with `articleSection: "Weekly Digest"`

## Requirements

### Functional Requirements

**Critical Fixes**:
- **FR-101**: Publisher Agent MUST call GitHub Contents API (`PUT /repos/{owner}/{repo}/contents/{path}`) to create real MDX files
- **FR-102**: Publisher Agent MUST support `HUMAN_REVIEW_MODE=true` (creates PR) and `HUMAN_REVIEW_MODE=false` (direct commit)
- **FR-103**: SBP Scraper MUST fetch PKR rates against USD, EUR, GBP, AED, SAR, CNY from SBP EasyData API or website
- **FR-104**: PBS Scraper MUST fetch weekly SPI data for at least 8 food commodities from pbs.gov.pk
- **FR-105**: Commodities Scraper MUST fetch petrol/diesel/kerosene prices from OGRA or PSO, and gold price from PMEX or Business Recorder RSS
- **FR-106**: Mandi Table component MUST fetch live data from `/api/commodities` D1 endpoint instead of using hardcoded mock data
- **FR-107**: System MUST expose `GET /api/exchange-rates`, `GET /api/commodities`, `GET /api/gold-price` Cloudflare Function endpoints reading from D1

**Tool Requirements**:
- **FR-201**: Electricity Bill Calculator MUST support all 9 DISCOs with their current NEPRA-approved residential tariff slabs
- **FR-202**: Electricity Bill Calculator MUST calculate: energy charges, fixed charges, GST (17%), fuel price adjustment (FCA), and total bill
- **FR-203**: Electricity Bill Calculator MUST support consumer types: Residential, Commercial, Agricultural
- **FR-204**: Zakat Calculator MUST calculate Nisab threshold in PKR from live silver price (D1) using 612.36 grams silver standard
- **FR-205**: Zakat Calculator MUST support Gold Nisab toggle (87.48 grams gold equivalent)
- **FR-206**: Zakat Calculator MUST accept: cash savings, gold weight/value, silver weight/value, business inventory value, receivables, and liabilities
- **FR-207**: Loan/EMI Calculator MUST support two financing modes: **Conventional** (standard EMI compound interest formula: `EMI = P × r × (1+r)^n / ((1+r)^n - 1)` where r = monthlyRate, n = tenureMonths) and **Islamic/Murabaha** (flat cost-plus: `totalProfit = P × annualRate% × tenureYears; monthlyInstallment = (P + totalProfit) / tenureMonths`). UI must label "Profit Rate" not "Interest Rate" in Islamic mode.
- **FR-208**: Loan/EMI Calculator MUST generate a 12-month amortization schedule with opening balance, installment, principal, profit/interest, and closing balance
- **FR-209**: Loan/EMI Calculator MUST support lump-sum prepayment simulation
- **FR-210**: Currency Converter MUST fetch live PKR rates from `/api/exchange-rates` D1 endpoint
- **FR-211**: Currency Converter MUST display a 30-day trend chart using D1 historical exchange rate data
- **FR-212**: Currency Converter MUST support: USD, EUR, GBP, AED, SAR, CAD, AUD, CNY, JPY, and RON (Romanian Leu for diaspora)
- **FR-213**: Salary Slip Generator MUST generate a downloadable A4-format PDF using pdf-lib.js (client-side only)
- **FR-214**: Salary Slip Generator MUST auto-calculate monthly advance income tax from TaxCalculator slabs
- **FR-215**: Salary Slip Generator MUST save templates to `localStorage`
- **FR-216**: Gold Investment Calculator MUST fetch current gold price from D1
- **FR-217**: Gold Investment Calculator MUST show projected value at configurable price change scenarios (+10%, +20%, +50%, -10%, -20%)
- **FR-218**: Inflation Impact Calculator MUST use CPI data from D1 (sourced from PBS)
- **FR-219**: Inflation Impact Calculator MUST support base years from 2015 to present
- **FR-220**: EOBI Calculator MUST use current EOBI pension formula and minimum wage as defined in law
- **FR-221**: Property Stamp Duty Calculator MUST support Punjab, Sindh, KPK, Balochistan, ICT with province-specific rates
- **FR-222**: Property Stamp Duty Calculator MUST differentiate filer/non-filer tax status via a self-declared toggle: "I am an Active Tax Filer (FBR)" — no CNIC input or external API verification. Calculator MUST display a note: "Filer status is self-declared. Verify your status at fbr.gov.pk/atl before completing any transaction."
- **FR-223**: Remittance Calculator MUST use live exchange rates from D1 for all supported currencies
- **FR-224**: Remittance Calculator MUST support: GBP, EUR, USD, AED, SAR, CAD, AUD with PKR output

**Agent Enhancement Requirements**:
- **FR-301**: Analyst Agent (Agent B) MUST call Groq API with `llama-3.3-70b-versatile` model for content generation
- **FR-302**: Analyst Agent MUST fall back to OpenAI `gpt-4o-mini` when Groq is unavailable
- **FR-303**: Analyst Agent MUST store `generated_by` field in D1 `market_insights` table for every insight
- **FR-304**: LLM-generated content MUST pass validation: word count > 250, contains ≥1 numeric figure from scraped data, no placeholder text
- **FR-305**: Agent D (Social Publisher) MUST post to Twitter/X after each successful Agent C commit. Posts MUST be English-primary with key numeric figures labelled in Urdu script (e.g., "Dollar rate: ڈالر ریٹ 278.50 — up 0.4% today"). Full RTL Urdu posts are prohibited due to Twitter rendering inconsistencies
- **FR-306**: Agent D MUST post to Telegram channel after each successful Agent C commit. Telegram posts MUST follow the same English-primary + Urdu-labelled-figures convention as Twitter posts. Full Urdu body text is permitted in Telegram only if the LLM generates it without prompting, but English remains the canonical language for all generated content
- **FR-307**: Agent D MUST support `TWITTER_ENABLED` and `TELEGRAM_ENABLED` feature flags
- **FR-308**: Agent D MUST queue failed social posts in KV with retry_at timestamp (retry window: 1 hour)
- **FR-309**: Weekly Economic Digest MUST be triggered by the existing 6-hour cron Worker (`0 */6 * * *`); the Worker MUST check if current time falls within Monday 08:00–14:00 PKT (UTC 03:00–09:00) and, if so, append a digest generation stage after the standard Scraper→Analyst→Publisher flow. No separate cron entry in `wrangler.toml` is required
- **FR-310**: Weekly Digest MUST include: top mover of the week, week-over-week comparison table, and LLaMA 3 generated economic context

## Key Entities

- **Market Insight (Enhanced)**: Adds `generated_by` (LLM model), `content_category` (market_insight/weekly_digest/budget_alert/policy_update), `twitter_post_id`, `telegram_message_id` to existing entity.

- **Electricity Tariff Slab**: Represents a single consumption slab for a DISCO. Attributes: disco_id (LESCO/HESCO/etc.), consumer_type (residential/commercial/agricultural), from_units, to_units (or null for final slab), rate_per_unit (PKR), fixed_charge (PKR/month), fuel_adjustment_charge (PKR/unit, updated monthly by NEPRA).

- **Zakat Calculation**: Represents a user's zakatable wealth assessment (ephemeral, never stored server-side). Attributes: cash_savings, gold_grams, silver_grams, business_inventory, receivables, liabilities, nisab_basis (silver/gold), calculated_zakat, nisab_met (boolean), calculation_date.

- **Loan Schedule**: Represents a loan amortization (ephemeral, client-side only). Attributes: principal, down_payment, annual_rate, tenure_months, financing_mode (conventional/islamic), monthly_installment, total_interest, total_payable, amortization_rows[].

- **Newsletter Subscriber**: Represents an email subscriber. Attributes: email, confirmed (boolean), subscribed_at, unsubscribed_at (nullable), confirmation_token (HMAC-SHA256).

- **Agent Log**: Represents an audit record for each agent workflow execution stage. Attributes: workflow_id, stage (scraper/analyst/publisher/social), status (running/completed/error/skipped), message, duration_ms, created_at.

- **Social Post**: Represents a published social media post linked to a market insight. Attributes: insight_id, platform (twitter/telegram), post_id, content, posted_at.

## Edge Cases

- **Groq rate limit exhausted**: Analyst Agent MUST fall back to OpenAI for that single request and log the fallback. If both are unavailable, the workflow logs an error and skips content generation for that cycle — it does NOT halt data collection.

- **DISCO tariff slab not found for entered units**: Electricity Calculator MUST display: "Units entered exceed our tariff data. Please verify your DISCO's current tariff order." and show the last known slab rate.

- **Zakat calculation with negative net wealth** (liabilities > assets): System MUST clearly state "No Zakat is due" and explain why, without showing negative Zakat amounts.

- **Loan calculator with 0% interest rate (interest-free loan)**: System MUST handle this as valid input — monthly installment equals principal divided by tenure months.

- **Currency converter when D1 has no data for selected currency**: System MUST display: "Rate not available — data is being updated" and offer a fallback manual rate input field.

- **PWA service worker update conflict** (old cache vs new deployment): Service worker MUST use a versioned cache name (e.g., `pakecon-cache-v2`) and delete old caches on activation.

- **GitHub API token expiry during Agent C publish**: Publisher MUST catch 401 response, log "GITHUB_TOKEN_EXPIRED: manual renewal required" to D1 agent_logs, and disable auto-publishing (set KV flag `publisher_disabled=true`) until manually reset by operator.

- **Newsletter subscriber confirms twice**: The confirmation endpoint MUST be idempotent — second confirmation returns success without creating duplicate records.

- **Agent D Twitter post with special characters** (Urdu text, currency symbols): Twitter post content MUST be sanitized to ASCII-safe characters + standard emoji; Telegram may retain full Unicode.

## Implementation Notes

### MDX Content Collection Frontmatter Schema

All insight MDX files committed by Agent C MUST use exactly this frontmatter structure:

```yaml
---
title: string          # SEO-optimised headline (max 70 chars)
pubDate: YYYY-MM-DD    # ISO date of publication
summary: string        # 1–3 sentence summary for AI Overviews / card UI
category: string       # One of: market_insight | weekly_digest | budget_alert | policy_update
source: string         # Primary official source URL cited (SBP / PBS / FBR / PMEX)
delta: number          # Percentage change that triggered this insight (0 for digests)
---
```

- Additional fields (`author`, `tags`, `relatedTool`, social post IDs) are stored in D1 `market_insights`, NOT in MDX frontmatter
- The Astro Content Collection schema (`src/content/config.ts`) MUST validate these 6 required fields with Zod; any MDX missing a required field MUST fail the build
- `category` values are an enum — addition of new categories requires both spec amendment and schema update
